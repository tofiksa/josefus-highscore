mvn test

## Required Environment Variables

```
SUPABASE_JOSEFUS_DB_URL     # PostgreSQL JDBC connection URL
SUPABASE_JOSEFUS_DB_USER    # Database username
SUPABASE_JOSEFUS_DB_PWD     # Database password
JOSEFUS_JWT_SECRET          # Base64-encoded HS256 secret (shared with josefus-auth, used for JWT validation)
```

## Architecture

Spring Boot 3 REST API — **game and highscore backend only**. Authentication, user registration, and token generation are handled by the separate `josefus-auth` Kotlin service.

- **`controller/`** — REST endpoints. `GameController` (score submission and ranking), `UserInfoController` (game history and user rank).
- **`service/`** — Business logic. `JwtService` handles JWT validation only (no token generation). `RankService` computes top-10 rankings using parallel streams. `GameService` manages game creation and score submission. `ScoreService` computes per-user score stats.
- **`register/`** — JPA repositories (Spring Data). Named `*Register` rather than `*Repository`.
- **`model/`** — JPA entities mapped to the `josefushighscore` schema in PostgreSQL. `User` entity is kept read-only for FK relationships (Game→User).
- **`dto/`** — Request/response DTOs: `GameDto`, `ScoreDto`.
- **`exception/`** — Custom exception classes plus `GlobalExceptionHandler` (`@RestControllerAdvice`) that standardizes error responses.
- **`configure/`** — Spring Security config, JWT filter, CORS properties, and Flyway/JPA configuration.

## Relationship with josefus-auth

This service shares the same Supabase PostgreSQL database and JWT secret with `josefus-auth`. The auth service generates JWT tokens; this service validates them. Both use the `josefushighscore` schema. The `User` entity here is read-only — user management (registration, password, sign-in tracking) is owned by josefus-auth.

## Security Model

JWT-based stateless auth. `JwtAuthenticationFilter` validates `Authorization: Bearer <token>` on every request and sets the `SecurityContext`. Tokens are generated by `josefus-auth` (1h access, 7d refresh). Method-level security uses `@PreAuthorize`/`@Secured`. Score ownership is enforced in the service layer before writes.

## Database

PostgreSQL (Supabase), schema `josefushighscore`. Flyway migrations run automatically on startup from `src/main/resources/db/migration/`. Migration files follow the naming `V{major}_{minor}_{patch}__{description}.sql`. Tests use an H2 in-memory database.

**Note:** Some legacy migrations (V0_0_1, V0_0_2, V0_1_1–V0_1_8) create/alter user-related tables. These ran before auth was extracted to josefus-auth and remain in the migration history. User table migrations are now owned by josefus-auth (separate Flyway history table: `flyway_schema_history_auth`).

## API Documentation

Swagger UI is available at `http://localhost:8080/swagger-ui/index.html` when running locally. OpenAPI spec at `/v3/api-docs`.

## Deployment

Deployed to Heroku. The `Procfile` defines the web process:
```
web: java -Dserver.port=$PORT $JAVA_OPTS -jar target/josefus-highscore.jar
```
`system.properties` pins the Java runtime to Zulu JDK 24.
